# 启动与时序审计报告

## 范围

检查了 Wails 后端启动、React 首次挂载、启动时渐进扫描、手动和定时刷新、单仓操作、Git 子进程环境，以及终端事件订阅。

## 总评

渐进扫描的方向是对的：仓库列表可以先出来，不必等所有 Git 状态和历史都读完。当前问题不在“并发本身”，而在两套调度彼此独立、没有结果归属和版本规则。共发现 3 条 P1：它们会造成旧结果回写、请求配置串用，或让队列永久等在一个无超时的 Git 进程上。因此建议做一次围绕“快照任务生命周期”的小型重构，而不是推倒重写。

## 下一步建议

- **P1 本迭代修**：先建立统一的快照任务协调器，再把代理和超时改成请求参数向下传递。
- **重构价值**：从“哪个请求最后返回就写哪个”变为“只有当前任务允许写回”；能消除偶发回退、减少后续靠补丁修顺序的成本，并让卡住的 Git 调用有明确结束条件。
- **不建议**：不需要替换 Wails、React 或把所有扫描改成串行。保留启动时每仓并发，只要纳入同一生命周期和版本控制即可。
