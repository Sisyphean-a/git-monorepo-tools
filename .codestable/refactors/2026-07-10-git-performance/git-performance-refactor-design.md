# Git 性能重构方案

## 1. 本次范围

- 用户已选择 scan 中的 #1 至 #5。
- 不改变 Git 命令的成功、失败或跳过语义；不增加超时、静默回退或数据截断。
- 批量同步仍显示所有目标仓库的结果；提交详情仍可从历史页读取。

## 2. 前置依赖

- ADR 001 规定交互、后台和按需详情的边界。
- 现有 `GetRepoHistory` 与 `GetCommitDetail` 是历史详情的读取接口。
- 现有单仓 `RepoSnapshotUpdate` 用于回写批量中实际发生变化的仓库。

## 3. 执行顺序

1. 使用 M-L4-03：从基础快照移除提交历史，历史页激活时请求首批摘要；日志命令改为摘要输出。
2. 使用 M-L4-03：未跟踪目录在基础快照中作为单个路径保留，不再递归列文件或读取文件内容。
3. 使用 M-L4-06：拆分后台刷新和交互任务通道；交互任务开始后，较早的后台结果不得写回。
4. 使用 M-L4-02：以轻量状态预检生成批量计划，按 `concurrency` 并行执行；仅回读 `pulled` 或 `pushed` 的仓库并合并更新。
5. 运行 Go 与前端验证，完成独立代码审查前不提交。

## 4. 风险与看点

- 后台刷新与交互并行时必须以代次防止旧快照覆盖新结果。
- 批量并发不能打乱结果展示顺序，也不能超过用户配置的并发数。
- 目录不再展开后，目录本身仍须能作为 `git add -- <path>` 的有效输入。
