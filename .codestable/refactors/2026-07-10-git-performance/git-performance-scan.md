# Git 性能重构扫描

## 总览

- 扫描范围：`snapshot/` 的 Git 读取与批量操作、`src/app/` 的快照协调和历史/日志界面，以及对应测试。
- 发现 5 条优化点：性能 5。
- 按风险：低 1 / 中 3 / 高 1。
- 建议先做：#1、#2、#5，它们有现成的分页历史接口和单仓回包机制。
- 建议慎做：#3、#4，它们涉及异步写回顺序和并发 Git 操作。
- 前置检查：跨模块边界已由 ADR 001 明确；用户已明确选择全部处理。

## 已选项

- [x] #1 暂存后停止读取提交历史
  - **位置**：`snapshot/service.go`、`snapshot/git.go`、`src/app/components/repo-history-tab.tsx`
  - **分类**：性能
  - **现状**：每次仓库快照都会读取最近 51 个提交及其 numstat，即使历史页未打开。
  - **问题**：单仓回包额外执行一次与当前操作无关的 Git 日志读取。
  - **建议**：基础快照不携带历史；历史页激活后通过现有分页接口读取首批摘要。
  - **建议映射的方法**：M-L4-03
  - **风险**：中（历史页需正确显示首次加载与切换仓库）。
  - **验证**：Go 单测、历史页逻辑测试、前端构建。
  - **范围**：约 5 文件。

- [x] #2 未跟踪目录停止递归统计
  - **位置**：`snapshot/git.go`
  - **分类**：性能
  - **现状**：未跟踪目录会递归枚举全部子文件，并逐个读取内容统计行数。
  - **问题**：目录大小决定刷新时间，时间和内存都随子文件数线性增长。
  - **建议**：基础快照只保留 Git 返回的目录路径；目录内容不在刷新路径展开。
  - **建议映射的方法**：M-L4-03
  - **风险**：低（暂存目录仍传给 `git add -- <directory>`）。
  - **验证**：新增目录测试，确认不再读取目录中子文件且目录可暂存。
  - **范围**：约 3 文件。

- [x] #3 交互操作绕过后台刷新队列
  - **位置**：`src/app/snapshot-coordinator.ts`、`src/app/use-snapshot-refresh.ts`
  - **分类**：性能
  - **现状**：远端刷新和暂存共用一个先进先出的队列。
  - **问题**：正在执行的远端刷新会让本地暂存无法开始。
  - **建议**：后台刷新与交互操作使用独立串行通道；旧后台结果按代次丢弃，不能覆盖较新的交互回写。
  - **建议映射的方法**：M-L4-06
  - **风险**：高（必须保证刷新与交互结果的写回顺序）。
  - **验证**：协调器时序单测覆盖并行刷新、立即交互和旧结果丢弃。
  - **范围**：约 4 文件。

- [x] #4 批量同步使用轻量预检和受控并发
  - **位置**：`snapshot/operations.go`、`snapshot/snapshot_workers.go`
  - **分类**：性能
  - **现状**：批量操作前后各构建一次完整快照，Git 操作逐仓串行。
  - **问题**：仓库数量增长时固定增加两轮详情读取和串行网络等待。
  - **建议**：用状态预检构建批量计划，按配置并发执行，并只回读实际执行成功的仓库。
  - **建议映射的方法**：M-L4-02
  - **风险**：高（结果顺序、并发上限与回写完整性）。
  - **验证**：Go 单测覆盖并发上限、跳过项和目标仓库回读；前端合并测试。
  - **范围**：约 6 文件。

- [x] #5 日志窗口改为提交摘要
  - **位置**：`snapshot/operations.go`
  - **分类**：性能
  - **现状**：日志窗口每次读取 10 个提交的完整 `--stat` 文本。
  - **问题**：大提交会产生大量无须立即展示的文本和跨进程传输。
  - **建议**：日志接口只返回提交摘要；文件统计继续通过历史页中的按提交详情读取。
  - **建议映射的方法**：M-L4-03
  - **风险**：低（日志窗口不再承担提交详情展示）。
  - **验证**：Go 单测断言日志命令不含 `--stat`，前端日志测试。
  - **范围**：约 3 文件。
