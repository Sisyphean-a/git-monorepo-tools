# Iteration 001

## Current Understanding

当前自动刷新链路的主要问题不在仓库发现，而在每次全量扫描都会顺序执行每个 repo 的 `git status`、`git diff` 和 `git log`。前端虽然已有自动扫描防重入，但手动刷新、设置变更刷新和自动刷新之间仍会重复触发全量扫描。

## Implementation Approach

先用临时 profiling 把总耗时拆成“找 repo”和“构建 repo 快照”两段，确认热点在后者；随后利用已有但未接入的 `concurrency` 设置做后端并发快照构建，并在前端加入刷新请求合并与串行自动调度，减少重叠扫描。

## Changes This Iteration

- 在 `snapshot.Request` 和前端 `buildSnapshotRequest()` 中补齐 `concurrency` 字段，把设置面板已有的并发配置真正传到后端。
- 新增 `snapshot/snapshot_workers.go`，把 repo 快照构建拆成串行与并行两条路径；默认并发回退到 3，并按 repo 数量裁剪 worker 数。
- `snapshot/service.go` 改为在构建全量快照时按请求并发度调用并发 worker 池，而不是逐 repo 串行执行。
- 新增 `src/app/use-snapshot-refresh.ts`，把刷新请求合并成“一个进行中的扫描 + 最多一次排队重跑”，避免手动刷新、设置保存和自动刷新重叠触发重复全量扫描。
- `src/app/App.tsx` 去掉原先基于 `setInterval` 和 `autoScanRunning` 的扫描逻辑，改为通过新 hook 串行调度自动刷新。
