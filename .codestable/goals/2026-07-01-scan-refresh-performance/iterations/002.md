# Iteration 002

## Current Understanding

上一轮已经证明并发快照构建能显著缩短扫描时间，但默认档位仍停在 `3`。同机实测显示 `5` 档还能进一步降低全量扫描耗时，因此需要把“当前机器上的更优默认值”也纳入最终交付状态，而不是只保留可选配置。

## Implementation Approach

把后端默认并发回退值、前端请求构造回退值和前端默认设置同时切到 `5`，保证未显式配置时也能直接吃到更快的扫描速度；同时补一处刷新队列状态清理，避免请求异常后残留旧的进行中 promise。

## Changes This Iteration

- `snapshot/snapshot_workers.go`：默认并发从 `3` 提升到 `5`。
- `src/app/settings.ts`：前端默认 `gitBehavior.concurrency` 从 `3` 提升到 `5`。
- `src/app/api.ts`：Wails `SnapshotRequest` 的 `concurrency` 回退值从 `3` 提升到 `5`。
- `src/app/use-snapshot-refresh.ts`：为刷新队列补 `finally` 清理，确保请求失败后仍可发起下一次刷新。
- `frontend/wailsjs/go/models.ts`：随 `wails build` 生成的 Request 绑定同步包含 `concurrency` 字段。

## Verification Evidence

- `go test ./...`：passed
- `npm run build`：passed（`wails build`，生成 `build/bin/git-monorepo-tools.exe`）
- 默认并发切到 `5` 后，5 次连续全量扫描：`1506 / 1304 / 1660 / 1820 / 1530 ms`
- 同机显式并发对比仍成立：`concurrency=1 -> 4648ms`，`3 -> 1609ms`，`5 -> 1116ms`

## Problems Encountered

- 无新的技术阻塞。剩余阻塞仍是流程 gate：当前对话没有显式的 Task agent 授权，因此还不能完成独立 code review 与功能验收。

## Next Attempt

等待 owner 决定是否授权 Task agent 执行独立 code review 与功能验收；若授权，则先完成 gate，再决定是否关闭 goal。

## State Update

本轮把默认性能档位也调到了当前机器上的更优值，交付状态比上一轮更接近最终目标；goal 继续保持 `active`，等待 review/acceptance 授权。
