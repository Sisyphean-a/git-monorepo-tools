# 分离交互操作与后台 Git 刷新

## Context

当前仓库快照会在暂存等交互操作后同步读取提交历史和未跟踪目录中的所有文件；后台刷新又与交互操作共用一个先进先出的队列。批量同步和日志窗口同样会一次传输超出当前界面所需的数据。

这使本地 Git 操作的等待时间受历史大小、目录大小和远端网络影响，也让较早开始的后台结果有机会覆盖较新的交互结果。

## Decision

我们将读取路径分为三类：

- 交互操作只回读目标仓库的基础仓库快照，不读取提交历史，也不递归展开未跟踪目录。
- 提交历史、提交详情和未跟踪目录内容属于按需详情；只有用户打开历史页、日志页或展开目录后才读取。
- 后台刷新不再阻塞交互操作。交互操作保持自身的顺序；在它之后完成的旧后台刷新结果不得覆盖其回写。批量同步使用受配置限制的并发，并只回读实际执行过的仓库。

## Consequences

- 暂存、刷新和批量同步减少不必要的 Git 调用、文件系统读取与前端传输。
- 历史页、日志页和目录首次打开时会单独加载，并需显示加载或错误状态。
- 前端调度器必须区分交互与后台操作，并维护结果的新旧关系。
- 后端接口需要补充按需目录读取与轻量批量回写，但现有仓库状态和 Git 操作语义保持不变。

## Alternatives Considered

- 保持全量快照，只缩短 Git 命令超时：无法消除与数据规模成正比的读取成本，也会掩盖慢远端问题。
- 仅把后台刷新插到队尾：已执行的远端刷新仍会阻塞用户操作，不能解决最影响交互的等待。
- 为未跟踪目录和日志设置硬性条数或大小上限：会静默遗漏真实变更，违反完整展示与显式失败的要求。
