# 逐文件展示未跟踪变更

## Context

基础仓库快照曾将同一未跟踪目录聚合为一个变更项，以减少大目录扫描和前端传输。但 Git 客户端通常按文件展示变更；目录聚合会让变更数量与 VS Code 不一致，也无法为后续逐文件暂存、查看和改造提供稳定的数据粒度。

## Decision

仓库状态统一使用文件粒度。执行 `git status --porcelain=v1 -b --untracked-files=all`，每个未跟踪文件生成独立的 `FileChange`，仓库变更数、未暂存数和详情列表均按实际文件计数。

本决定仅取代 ADR 001 中“不递归展开未跟踪目录”和“未跟踪目录内容按需读取”的部分；交互与后台刷新分离、历史按需读取及结果时序约束继续有效。

## Consequences

- 变更列表和数量与 VS Code 等按文件展示的 Git 客户端保持一致。
- 后续逐文件暂存、丢弃、预览和筛选可以直接复用现有 `FileChange` 模型。
- 包含大量未跟踪文件的仓库会产生更多 Git 输出、文件统计和前端传输，刷新成本随实际文件数增长。
- 不设置静默条数上限；性能问题必须显式优化数据读取或交互，而不能遗漏真实变更。

## Alternatives Considered

- 保留目录聚合：刷新更轻，但展示口径不一致，且后续文件级操作需要再次拆分数据模型。
- 点击目录后按需展开：可以延后成本，但当前变更列表不是目录树，需要新增接口、加载状态和缓存一致性处理。
- 限制未跟踪文件数量：可以控制成本，但会静默遗漏变更，不符合完整展示要求。
